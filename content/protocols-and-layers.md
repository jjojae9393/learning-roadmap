## 프로토콜의 의미와 종류

프로토콜은 컴퓨터들이 통신할 때 지켜야 하는 **약속이자 규칙**입니다. 역할에 따라 종류가 나뉩니다.

- **HTTP / HTTPS** — 웹 콘텐츠를 주고받는 프로토콜
- **TCP / UDP** — 데이터를 어떻게 전달할지 정하는 전송 프로토콜
- **IP** — 경로를 정하는 프로토콜
- **DNS** — 주소를 변환하는 프로토콜

여기서 짚어둘 중요한 점은, **HTTP/HTTPS는 암호화 프로토콜이 아니라 웹 프로토콜**이라는 것입니다.
`HTTPS = HTTP + TLS` 이며, 실제 암호화를 담당하는 건 별도의 보안 프로토콜인 **TLS**입니다.

## 계층 구조 (TCP/IP 4계층 모델)

통신 과정을 여러 층으로 나누고 각 층에 역할을 하나씩 맡긴 구조입니다.
윗층은 아랫층에 일을 맡기고, 아랫층의 내부 처리는 신경 쓰지 않습니다.
위(사용자와 가까운 쪽)에서 아래(물리적 전송)로 갈수록 토대가 넓어지는 피라미드로 그려볼 수 있습니다.

데이터를 보낼 때는 **위에서 아래로** 내려가며 각 층이 정보를 덧붙이고,
받을 때는 **아래에서 위로** 올라가며 한 겹씩 벗겨 원래 내용을 복원합니다.

> 핵심은 HTTP, TCP, IP가 **경쟁 관계가 아니라 서로 다른 층에서 협력하는 관계**라는 점입니다.
> (OSI 7계층은 같은 개념을 더 세밀하게 나눈 모델입니다.)

## 암호화(TLS)의 위치

TLS는 전송 계층에서 실행되는 게 아니라, **전송 계층(TCP) 위, 애플리케이션 계층(HTTP) 아래**에 자리 잡고 동작합니다.
TCP가 깔아준 신뢰성 있는 통로 위에서 내용을 암호화합니다.

- 보낼 때: `HTTP → TLS(암호화) → TCP → IP`
- 받을 때: 그 역순

## 애플리케이션 계층의 정확한 의미

애플리케이션 계층은 '사용자에게 화면을 보여주는 단계'가 아니라,
**응용 프로그램끼리 무슨 내용을 주고받을지 정하는 규칙**입니다.

화면에 그려 보여주는 일은 브라우저 같은 프로그램이 하는 별개의 일이고,
애플리케이션 계층(HTTP)은 데이터를 주고받는 규칙까지를 담당합니다.
**데이터를 주고받는 규칙과 그것을 보여주는 방식이 분리돼 있는** 구조입니다.

## TCP는 어떻게 연결을 맺는가 (3-way handshake)

TCP 연결 과정은 흔히 **3-way handshake(3방향 핸드셰이크)**라고 부릅니다.
본격적으로 데이터를 주고받기 전에 양쪽이 "연결 준비됐나요?"를 서로 확인하는 절차입니다.

앞서 TCP는 신뢰성을 중시한다고 했죠. 그 신뢰성의 출발점이 바로 이 과정입니다.
무작정 데이터를 보내기 전에, 먼저 양쪽이 서로 통신할 준비가 됐는지 확인하고 연결을 맺습니다.
이름 그대로 세 번의 메시지가 클라이언트(요청하는 쪽, 예: 브라우저)와 서버(응답하는 쪽) 사이에 오갑니다.

**1단계 — SYN (클라이언트 → 서버)**

클라이언트가 서버에게 "연결하고 싶습니다"라고 요청을 보냅니다.
이때 보내는 신호를 SYN(synchronize, 동기화)이라고 합니다. 일종의 노크인 셈입니다.

**2단계 — SYN + ACK (서버 → 클라이언트)**

서버가 "요청 받았습니다(ACK). 저도 연결할 준비됐습니다(SYN)"라고 응답합니다.
받았다는 확인과 자신도 연결하겠다는 신호를 한꺼번에 보냅니다.

**3단계 — ACK (클라이언트 → 서버)**

클라이언트가 마지막으로 "서버의 응답도 잘 받았습니다(ACK)"라고 확인을 보냅니다.
이 세 번째 메시지가 도착하면 연결이 정식으로 맺어지고, 이제부터 실제 데이터를 주고받기 시작합니다.

비유하자면 전화 통화의 시작과 비슷합니다.
"여보세요?"(SYN) → "네, 들려요. 말씀하세요"(SYN+ACK) → "네, 저도 잘 들립니다"(ACK).
서로 목소리가 잘 들리는지 확인한 뒤에야 본 대화를 시작하는 것이죠.

> **왜 굳이 세 번이나 주고받을까요?** 양쪽 모두 보내고 받는 양방향이 다 정상인지 확인하기 위해서입니다.
> 2단계까지만 하면 서버는 클라이언트의 응답을 아직 못 받은 상태라 자기 쪽 송신이 제대로 가는지 확신할 수 없습니다.
> 세 번째 확인까지 있어야 양쪽 다 "내가 보낸 것도 잘 가고, 상대가 보낸 것도 잘 온다"는 걸 확신할 수 있습니다.

참고로 연결을 끝낼 때도 비슷한 절차가 있는데, 이때는 보통 네 번의 메시지가 오갑니다(**4-way handshake**).
양쪽이 각각 "이제 보낼 데이터 다 끝났습니다"와 "확인했습니다"를 주고받으며 깔끔하게 마무리합니다.

그래서 TCP 통신 전체를 큰 흐름으로 보면 이렇습니다.
먼저 3-way handshake로 연결을 맺고 → 그 위에서 (필요하면 TLS 암호화를 더해) 데이터를 패킷으로 주고받으며 확인·재전송하고 → 다 끝나면 handshake로 연결을 정리합니다.
앞서 본 내용들이 하나로 이어지는 셈입니다.
